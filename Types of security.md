# Виды безопасности

### Введение

Наша группа - команда QA, которая начинает тестирование новой функциональности в веб-приложении — "Безопасный чат для врачей и пациентов".<br> В чате передаются **конфиденциальные диагнозы, назначения и личные данные**.<br> Учитывая высокую критичность данных (персональные данные и медицинская тайна), необходимо провести предварительный анализ угроз безопасности.

Функциональности приложения: 

- личные чаты между врачом и пациентом;
- обмен текстовыми сообщениями и файлами (анализы, назначения);
- хранение истории переписки;
- авторизация пользователей (врач / пациент);
- доступ к чату через браузер и, возможно, мобильные устройства.

### Матрица рисков: таблица с объединенными рисками для каждого из принципов.

**Конфиденциальность**

| Риск | Описание | Причина проблемы безопасности |
| ---- | -------- | ----------------------------- |
| Доступ к чужим чатам                     | Пользователь может увидеть переписку другого пациента        | Отсутствие или ошибки в проверке прав доступа     |
| Перехват сообщений                       | Сообщения могут быть перехвачены в сети                      | Отсутствие или неправильная настройка HTTPS / TLS |
| Утечка данных через логи                 | Сообщения или диагнозы попадают в логи сервера               | Логирование чувствительных данных                 |
| Кража сессии                             | Злоумышленник получает доступ к чату через украденную сессию | Небезопасная работа с сессиями и cookies          |
| Доступ администраторов без необходимости | Технический персонал читает переписку                        | Не используется принцип минимальных привилегий    |

**Целостность**

| Риск | Описание | Причина проблемы безопасности |
| ---- | -------- | ----------------------------- |
| Подмена сообщений      | Сообщения могут быть изменены при передаче     | Нет проверки целостности данных                 |
| Изменение истории чата | Старые сообщения редактируются без уведомления | Отсутствие аудита и фиксации изменений          |
| Подмена файлов         | Пациент получает изменённый файл               | Отсутствие проверки хэшей файлов                |
| XSS-атака              | Вредоносный скрипт меняет содержимое чата      | Недостаточная валидация пользовательского ввода |
| Ошибки синхронизации   | Сообщения теряются или искажаются              | Проблемы с обработкой одновременных запросов    |

**Доступность**

| Риск | Описание | Причина проблемы безопасности |
| ---- | -------- | ----------------------------- |
| Отказ сервиса (DoS)               | Чат становится недоступным              | Отсутствие защиты от DoS/DDoS                      |
| Блокировка аккаунта               | Пользователь не может войти в систему   | Агрессивные или некорректные политики безопасности |
| Ошибки после обновлений           | Чат перестаёт работать после релиза     | Отсутствие тестирования и rollback-стратегии       |
| Перегрузка сервера                | Сообщения не доставляются вовремя       | Отсутствие масштабирования                         |
| Зависимость от сторонних сервисов | Чат не работает при сбое стороннего API | Нет резервных механизмов                           |

### Топ-3 критических риска по мнению нашей группы:

1. **Доступ к чужим чатам**

   - Самый критичный риск, так как приводит к прямому раскрытию медицинской тайны и персональных данных пациентов.

2. **Изменение медицинских данных в чате**

   - Подмена сообщений или файлов может привести к неправильному лечению и нанести вред здоровью пациента.

3. **Перехват сообщений при передаче**

   - При отсутствии корректного шифрования данные могут быть перехвачены, особенно при использовании публичных сетей.

### Рекомендации для разработчиков:

1. Доступ к чужим чатам:

   - Реализовать строгую серверную проверку прав доступа для каждого запроса;
   - Использовать принцип минимальных привилегий для пользователей и администраторов.

2. Нарушение целостности данных:

   - Использовать контроль целостности сообщений и файлов (например, хэши);
   - Ввести аудит изменений и запрет редактирования медицинских данных без фиксации.

3. Перехват сообщений:

   - Обязательное использование HTTPS с корректной настройкой TLS;
   - Запрет небезопасных протоколов и слабых алгоритмов шифрования.

Предварительный анализ показал, что для безопасного чата необходимо правильно настроить контроль доступа, защиту передаваемых данных и обеспечение целостности медицинской информации. Эти аспекты должны быть проверены уже на ранних этапах тестирования.
